# 機制規格：活網格版曼陀羅

這份文件說明資料模型與互動邏輯，render 的細節先不寫。整體的設計目標是讓這張圖如實反映你的實際狀態：有經營的地方亮、荒廢的地方暗。

**實作狀態**：目前 viewer 做了「全貌渲染」和「點格記一筆」。標 ⏳ 的部分（時間軸 pivot、封存與槽位釋放的 UI）還是規劃、尚未實作。小格的流動目前都靠手動編輯 `.md`（見 §3 末）。

## 1. 結構

- **8 個大格**：八塊固定的領域。範例用的是經濟、信仰、健康、工作事業、人際、嗜好、夢想、家庭伴侶，依九宮格位置 1–9 排列，中央（位置 5）留白。這些領域近乎永久、很少更動，是作者個人的分類，使用時請換成自己的。
- **每個大格內有 8 個槽**，放流動的子焦點（小格）。

幾何上沿用大谷翔平的 9×9 曼陀羅（Mandal-Art）：中央 3×3 是八塊領域，每塊領域再展開成自己的 3×3。

## 2. 大格與小格的差別

- **大格是固定的骨架。** 八塊領域是穩定的，設定一次後很少改。
- **小格是會流動的子焦點。** 它們表示「我在這塊領域裡現在在乎什麼」，它們的變化就是焦點變遷史。例如信仰這塊，可能某段時間是禱告與讀經，後來變成服事與懷疑，再後來是教會關係；依時間往下讀，就是屬靈焦點的移動。
- **父子關係**：小格屬於某個大格。如果大格被重新定義（很少發生），底下的小格需要重新整理。

## 3. 子焦點的生命週期

```
誕生 ──→ 保鮮 ──→ 枯萎 ──→ 封存（釋放槽位）
 │         │         │
 冒出新    持續註記   不理它
 焦點,     → 刷新     → 照時間戳
 佔空槽    last_tended  連續變暗
           → 保持明亮
```

- **誕生**：新焦點佔用一個空槽。
- **保鮮**：你為它新增記錄，`last_tended` 更新，格子變亮。
- **枯萎**：一段時間沒有記錄，格子依 `last_tended` 連續變暗，不是非亮即暗的二元狀態。
- **封存**（⏳ 目前沒有 UI，手動改 `.md`）：枯到底後釋放槽位給新焦點。封存不刪除資料，只是移出活躍區，保留變遷史。

一塊地同時活躍的子焦點上限是 8 個，因為一塊領域同時能專注的事情有限。資料層會檢查這點：活躍小格超過 8 個時會直接報錯，而不是默默忽略第 9 個。

目前小格的所有流動——新增、改名、封存——都直接編輯 Markdown 完成。viewer 只提供「看全貌」和「記一筆」兩個操作，沒有為這些流動做介面。這是為了讓介面保持簡單，不是功能未完成。

## 4. 標記現況，不是填滿

- 不需要為了補滿而硬擠內容，只更新真正有動靜的格子。
- 久未記錄的格子會自己淡成迷霧，這是一個誠實的訊號（這塊地這個面向最近沒有動靜，或在迴避），不是失職、待辦或該有罪惡感的事。
- 操作的心態是如實標記，而不是填滿。把荒地如實標成荒地，正是這個工具的重點。

## 5. 荒廢的判定

- 荒廢不是非亮即暗的二元狀態，而是由 `last_tended` 推算的連續程度（例如「信仰：已 5 個月沒有記錄」）。
- 改不改格子的名稱不是訊號，不改名是正常狀態。唯一的荒廢訊號是「多久沒有新記錄」。
- **大格層的荒廢**（用來鳥瞰、看出自己在冷落哪塊地）這樣算：

  > 大格的新鮮度 = 它底下所有活躍小格中，最近一筆記錄的日期。

  這裡不採用大格 frontmatter 上的 `last_tended`，因為那個值會被建檔日影響，可能讓一塊從沒真正經營過的地，只因為「昨天剛建檔」就顯得很新。

## 6. 底層資料模型：活網格加時間視角

資料不是每隔一段時間存一份快照，而是一張持續更新的網格：

- 每個格子帶 frontmatter：`大格`、小格的 `詞`、`born`、`last_tended`、`archived`（soft-delete）。
- 記錄是掛在格子上、帶日期的事件，一個格子可以有多筆不同日期的記錄。
- 「某段時間」是一個依日期過濾的視角，用來呈現那段時間哪些格子是亮的，而不是一種儲存單位。

這個模型讓幾個需求自然成立：

| 需求 | 如何達成 |
|---|---|
| 焦點變遷 | 亮點隨時間在格子間移動（新焦點變亮、舊焦點淡出），不需要 diff |
| 荒廢 | 格子依 `last_tended` 連續枯萎 |
| 時間軸 | 依日期過濾即可得到，不需額外結構 |
| 全貌 | 用「此刻」這個時間視角看整張網格 |
| 沒有填滿壓力 | 格子會自己保留到淡出，你只需要新增、記錄或忽略 |

子焦點不再活躍時不要真的刪除，否則會失去變遷史。改用淡出或封存（soft-delete，或 valid-from/to 的做法），時間視角才能照樣顯示它「曾經在某段時間是活躍的」。

## 7. 資料層與工具層

- **資料層**：純 Markdown 加 frontmatter，永久可讀、與工具無關。資料的延續性靠這一層，必須能獨立於任何工具存在。
- **工具層**：Astro renderer，可以丟掉。它故障或重寫都不應影響到資料。
- 兩種視圖：**全貌**（八塊領域並排呈現現況，✅ 已做）和**時間軸 pivot**（單一領域的長期趨勢，⏳ 尚未做）。pivot 之後需要補上，否則只看得到單一時間點的橫切面。

## 8. 留待實際使用後再決定的部分

以下幾點先不預先設計，等累積足夠的實際使用再定：

- 槽位釋放的門檻：枯多久才算封存。
- 用時間視角看某段時間時：顯示「那段時間有動靜的」還是「那段時間活躍的」格子。
- render 的視覺語言（地形、明暗、雜草等）。

這些用實際使用的結果來決定，而不是憑想像預先設計。
