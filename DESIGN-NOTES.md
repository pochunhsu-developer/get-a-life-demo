# 設計筆記

這份筆記記錄做這個人生盤點工具時的幾個設計決定。工具本身很小、也很個人，但其中幾個取捨應該能通用，整理出來給想做類似工具（PKM、習慣追蹤、自我盤點）的人參考。

更私密的動機和我自己的人生內容不在這個 repo，這裡只留可以遷移到別處的部分。

## 1. 先決定專案該多大，再寫 code

v1 把範圍限制在兩件事：看得到全貌、點一下記一筆。其他功能一律等到實際用過、確定真的不夠，再加。

比較理想的順序是：先用純文字手動記錄一陣子，確認自己真的會回來看，再投入寫 renderer。我自己沒做到這點——資料才開了一天就動手寫 viewer，原因是我喜歡看到做得好看的東西。正因為知道自己會這樣，才把這條寫下來提醒。

## 2. 資料用 Markdown + frontmatter，不用 JSON / DB

按重要性排列的理由：

- **工具無關、永久可讀**：就算將來沒有任何 runtime，純文字檔還是打得開。
- **可以手改**：不透過 app 也能直接編輯，app 只是加速工具，不是必經的入口。
- **適合版本控制**：每次更動都能用 git 留下歷史、可回溯。
- **資料與工具分離**：工具可以丟掉、重寫，刪掉整個 viewer 也不會影響到資料。

對一個打算自己長期使用的工具來說，這些好處比資料庫帶來的查詢便利更重要。

## 3. 標記現況，而不是填滿格子

久未記錄、已經淡成迷霧的格子，代表的是現況，不是失職，也不是待辦事項。它是一個誠實的訊號：這塊地最近沒有動靜，或在迴避。

所以這個工具是為「如實反映」設計，而不是為「填滿」設計。如果開始為了補滿格子而硬擠內容，這張圖就失去意義了。

## 4. 不刪資料，用 soft-delete 加時間視角

一個子項目不再活躍時不要真的刪掉，否則會失去它曾經存在過的歷史。改用 `archived` 標記讓它淡出即可。

「某段時間」不是一種儲存單位，而是一個依日期過濾的視角。資料只要帶著日期，「時間軸」和「焦點變遷史」就能直接從同一份資料算出來，不需要另外建表或做 diff。這是一種輕量的 event-sourcing 想法。目前的 viewer 還沒做出這個時間視角，但資料已經備好，需要時可以補上。

## 5. 讓它變成你會想打開的東西

對一個「只有自願回來用才有價值」的工具，外觀和使用意願其實是它能不能持續下去的關鍵之一：沒有人回來用，再好的設計也等於沒用。所以「好不好看、想不想打開」不只是裝飾。

這個工具用等高線地形當背景，是因為八塊領域本來就借用了「土地」的比喻：有在照顧的地比較亮，荒廢的地淡成迷霧，整張圖看起來像一片有人走過、也有荒地的地貌。背景的視覺和資料講的是同一個比喻，所以光是看著它，就會想起它在表達什麼。比起一個好看但與內容無關的背景，這樣的選擇多承擔了一層意義。

---

以上是模式，不是規則。它們是針對「一個人、長期、誠實面對自己」這個情境整理出來的。你的情況不同，看過一遍、留下對你成立的部分就好。
