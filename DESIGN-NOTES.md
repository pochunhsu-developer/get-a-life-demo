# 設計筆記：做一個「照妖鏡」型的個人工具

這是我替自己做人生盤點工具時，沿路收斂出來的幾個取捨。工具本身很小、很個人，
但底下這幾個決定我覺得能通用，抽出來記在這——給任何想做 PKM / 習慣追蹤 / 自我盤點工具的人參考。

（更私密的動機與我自己的人生內容不在這個 repo；這裡只留可遷移的模式。）

## 1. 先決定專案該多大，再寫 code

- **v1 硬封範圍**：這個工具只做兩件事——「看得到全貌」＋「點一下記一筆」。其餘一律留到「用了真的不夠」再說。
- **紀律（我自己破了戒）**：照理該先用純文字手動跑一陣子、確認你真的會回來看，**再**投資寫 renderer。我沒做到——資料才開一天就動手寫 viewer 了。主因很誠實：我就是喜歡看到漂亮的東西。正因為破過戒，這條才更要寫下來。

## 2. 資料用 Markdown + frontmatter，不用 JSON / DB

理由，按重要性：

- **工具無關、永久可讀**：十年後沒有任何 runtime，純文字照樣打得開。
- **人可手改**：不靠 app 也能直接編輯，app 只是加速器不是守門員。
- **git diff 友善**：每次改動天然有版本、有歷史、可回溯。
- **資料與工具徹底分離**：工具可拋棄、可重寫，`rm -rf` 掉整個 viewer 都不該動到資料一根寒毛。

對「給自己長期用」的工具，這條的價值遠大於 DB 帶來的查詢便利。

## 3. 標記，不是填滿

久未照料、自己枯成迷霧的格子，不是失職、不是待辦、不該有罪惡感——它是**誠實的訊號**（這塊我最近沒動靜／在迴避）。
工具要為「誠實」設計，而不是為「填滿」設計。一旦你開始為了補滿格子而硬擠內容，這面鏡子就失準了。

## 4. 不刪資料，用 soft-delete + 時間鏡頭

一個子項「死掉」時不要真刪（會丟失變遷史），用 `archived` 旗標淡出。
「某段時間」不是一個儲存單位，而是一個**濾日期的時間鏡頭**——於是「時間軸 / 焦點變遷史」幾乎免費**可做**，不用額外建表、不用 diff。這是一種輕量的 event-sourcing 心態。（坦白說：我自己的 viewer 還沒把這個時間視圖做出來，但資料模型已經把它備好了——這正是「純文字資料層」的紅利。）

## 5. 讓它變成你想打開的東西——而且讓美學承載隱喻

這條最容易被當成虛榮，但對「靠自願回來才有價值」的工具，它其實是存活機制：沒有回來的人，再好的設計都是死的。所以「好不好看、想不想打開」不是裝飾，是它還活著的條件之一（這也是上面 §1 那句「我就是喜歡看到漂亮的東西」的正當版本）。

但別停在「漂亮」。我刻意選了**等高線地形**當佈景，因為八塊地本來就是「土地」的隱喻——照料的地明亮、荒廢的地枯成迷霧，整張圖像一片可走可荒的地貌。**美學跟資料講同一個隱喻**，所以光是看著它，就在提醒我它想說什麼；它比一個漂亮但無意義的主題多做了一倍的工。

代價要老實講：這正是「A 吃 B」最會躲的地方。同樣是調佈景，可能在服務「我更願意回來面對自己」，也可能只是「我在用刻畫面逃避面對自己」——兩者長得一模一樣，只有你自己分得出來。判準就一句：**它讓你更常打開來誠實記錄，還是更少？**

---

這些是模式，不是教條；它們是為「一個人、長期、誠實面對自己」這個情境調出來的。

你的情境不同，照著想一遍、留下對你成立的就好。
